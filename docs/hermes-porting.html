Dual-runtime guide

Port a plugin to Hermes

Make a nautilai skill run in Hermes Agent as well as Claude Code — from this one repo, without changing a single Claude-facing file. Hermes support is purely additive; a Claude regression is never an acceptable price for it.

The one mental model

Read this before touching anything

Hermes installs the skill directory and nothing else. Claude Code loads the whole plugin — so a plugin's root-level scripts/ and templates/ (siblings of skills/) never reach Hermes, and a skill whose workflows reference them would install broken.

The fix is not to move those resources — that would rewrite dozens of path references across workflows and tests, real regression surface on the runtime that already has users. Instead we mirror them into the skill dir for Hermes, keep the plugin-root copies as the source of truth for Claude, and add one additive adapter section to SKILL.md that tells each runtime which path to use.

The invariant Your PR diff shows no changes under <plugin>/scripts/**, templates/**, tests/**, or .claude-plugin/**. The only edit to a Claude-loaded file is the additive Resource paths section in SKILL.md. Prove it with the diff.

Step 0 — Can it port at all?

Two disqualifiers, checked first

✗
Does the plugin's value depend on subagents?
Hermes has no subagent primitive. If fan-out and worktree isolation are the plugin (autodev), a port is a hollow shell — keep it Claude-only. Skills that only optionally fan out for review (review-plan, dep-review, pr-comment-review) already document an inline fallback; that documented path is their Hermes behavior.
✗
Does a bundled script provision tooling?
Hermes security-scans every installed skill, and for a community source any CAUTION verdict blocks the whole bundle. A script that runs npm install / pip install, or reads ~/.ssh/, scores MEDIUM supply_chain + HIGH exfiltration. Keep such logic in one script so it can be excluded cleanly, and make that workflow Claude-only.
✓
Otherwise — it ports.
A single skill whose scripts only read/analyze, plus its workflows and templates, is a clean candidate. Follow the six steps below.

The port, step by step

Six additive edits, one script, one verify pass

  1. Register the skill in the mirror script

    Add a plugin:skill pair to PAIRS in hermes/sync-resources.sh. This is the list of skills whose plugin-root scripts/ and templates/ get mirrored into the skill dir.

    hermes/sync-resources.sh
    # plugin:skill pairs whose resources are mirrored into the skill dir.
PAIRS=("commitcraft:commitcraft" "myplugin:myskill")
DIRS=("scripts" "templates")
  2. Exclude any provisioning script

    If the plugin has a script that installs packages, mutates git config, or reads keys (see Step 0), add it to EXCLUDE so it never ships to Hermes. Paths are relative to the plugin dir; a directory entry excludes everything under it.

    hermes/sync-resources.sh
    # Files NOT mirrored to Hermes — provisioning logic the scanner blocks.
EXCLUDE=("scripts/commitcraft-setup.sh" "templates")

    Why: the security scanner is usually right — provisioning tooling is installing packages. Never tell users to --force past it. Ship a bundle that passes; document the manual steps in the README.

  3. Regenerate the mirror

    Run the script. It copies each DIRS entry into <plugin>/skills/<skill>/, minus the exclusions. The mirror is generated and inert to Claude — no Claude-loaded file references it — and CI gates it against drift. Never hand-edit it; edit the plugin-root copy and re-run.

    $ bash hermes/sync-resources.sh

    Commit the generated files alongside your source changes — CI runs hermes/sync-resources.sh --check and fails if the mirror is stale.

  4. Add the Resource-paths adapter to SKILL.md

    This is the only edit to a Claude-facing file. One section, one line per runtime. Each runtime substitutes only its own token and leaves the other as literal text — so the line that resolved to a real absolute path is the one to follow. Self-disambiguating; nothing to guess.

    skills/<skill>/SKILL.md — additive section
    ## Resource paths

- **Claude Code:** `${CLAUDE_PLUGIN_ROOT}/scripts/<name>.sh`
- **Hermes:** `bash ${HERMES_SKILL_DIR}/scripts/<name>.sh`

Never substitute a token yourself; never fall back to a
relative path. Everywhere a workflow says
`${CLAUDE_PLUGIN_ROOT}/scripts/<name>.sh`, run
`bash ${HERMES_SKILL_DIR}/scripts/<name>.sh`.

    Why the workflows stay untouched: workflow files are never edited (the invariant), so they name only the Claude path. The adapter carries the translation. This is safe because it's the same mechanism Claude already relies on — ${CLAUDE_PLUGIN_ROOT} was never a shell variable; the model resolves it from context.

  5. Invoke bundled scripts via bash on the Hermes line

    Already shown above, but it's the single most common footgun, so it gets its own step: Hermes strips the executable bit on install. A file committed 100755 lands 644, and a direct ./script.sh call returns Permission denied. Keep Claude's direct invocation; prefix only the Hermes line with bash.

  6. State the limits — in SKILL.md, the README, and llms.txt

    Where a workflow is excluded (Step 2) or a subagent path is unavailable (Step 0), the SKILL.md adapter must say so explicitly and tell the agent not to improvise an equivalent — an agent asked to "set up commitlint" with no script will happily start installing packages itself.

    Scanner trap — keep SKILL.md payload-free SKILL.md is shipped and scanned. It carries instructions, not rationale. Never quote a package-install command, a key path, or an attack string to explain why something was excluded — a static scanner matches patterns, not intent, and prose describing a blocked script can re-trigger the very findings that blocked it. A DANGEROUS verdict can't be bypassed even with --force. Name the category of thing; put the why in the README and the convention doc.

    Then complete the paperwork: the plugin's README states all five headings — shared behavior · Claude Code invocation · Hermes invocation · runtime-specific limitations · update behavior — and the plugin is added to docs/llms.txt with the runtimes it supports.

Verify before you ship

A green mirror is necessary, not sufficient

$ bash hermes/sync-resources.sh --check
$ claude plugin validate ./<plugin> --strict
Probe with the real artifact The security scanner judges the payload, not a mechanism. A trimmed-down probe will pass where the real script is BLOCKED — a ~30-line stand-in returned SAFE while the real 1,500-line provisioner scored 12 findings. If the payload is what gets judged, probe with the real file in a throwaway repo, or accept that the risk is still open. And publishing a public repo is publishing: skills.sh auto-indexes any public repo containing a SKILL.md — no tap, no opt-in, no way to hide.

Gotchas the docs got wrong

Verified against the binary, not the vendor docs

Treat published Hermes docs as a hypothesis, not a spec. These claims did not hold on Hermes v0.18.2:

"hermes skills tap add <repo> indexes a GitHub repo"
Registers a row and indexes nothing — the github source is permanently skipped as "slow". Skills resolve by name at any depth instead; no tap, no skills/ root needed.
"version, author, license are required frontmatter"
Not enforced — a skill without them installs fine.
"Updates need a version bump"
Hermes detects drift by content, not version — upstream fixes arrive via hermes skills check then hermes skills update with no bump. Users update; you don't re-tag.

Ship checklist

Every box before the PR

  • Diff shows zero changes under scripts/**, templates/**, tests/**, .claude-plugin/**.
  • PAIRS updated; provisioning scripts in EXCLUDE; bash hermes/sync-resources.sh re-run and the mirror committed.
  • SKILL.md has an additive Resource paths section — Hermes line prefixed with bash, excluded workflows named with a no-improvise instruction, and no quoted payloads.
  • README states the five headings; the plugin is in docs/llms.txt with its runtimes.
  • sync-resources.sh --check and claude plugin validate --strict both pass.

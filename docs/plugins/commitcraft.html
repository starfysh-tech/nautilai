commitcraft

AI git workflow toolkit — conventional commits, issue validation, PR creation, and release guidance in one skill.

Install

/plugin install commitcraft@nautilai

What it does

CommitCraft handles the git workflow around a change — not just the commit message. One skill routes to six workflows: conventional commits with individually staged files, push with issue tracking, PR creation with an AI-generated description, semantic-version releases, interactive tooling setup, and a configuration check. It also triggers from plain language — "commit my changes", "open a PR", "cut a release" — without the slash form.

Commands / skills

  • /commitcraft commit

    Stages files individually, generates a conventional commit message, and handles pre-commit hooks.

    "commit my changes", "commit this"
  • /commitcraft push

    Full commit + push with issue validation, branch tracking, and post-push issue comments.

    "push my changes"
  • /commitcraft pr

    Creates a PR with an AI-generated description, issue linking, and draft support.

    "open a PR", "create a pull request"
  • /commitcraft release

    Guides a semantic-version bump via release-please when it's functional, or a manual tag workflow when it isn't.

    "cut a release", "write release notes"
  • /commitcraft setup

    Interactive, chat-driven setup across eight components: commitlint, gitleaks, pre-commit, signing, release-please, CI, issue tracker, and branch protection.

    "set up commitcraft"
  • /commitcraft check

    Validates installed tooling and reports configuration status.

    "check my commit tooling"

How it works

The skill reads skills/commitcraft/workflows/<argument>.md and follows it; an unrecognized argument is rejected with the list of valid subcommands rather than defaulting to commit. Workflows shell out to bundled scripts via ${CLAUDE_PLUGIN_ROOT}/scripts/…, so paths resolve correctly regardless of where the plugin cache lives.

The release workflow defers to release-please only when it's actually functional — a detection script distinguishes a working release-please from one that's present but neutered (skip flags, no contents: write, or stuck at 0.0.0), falling back to a manual tag/release path with a one-line reason rather than dead-ending on a release PR that will never arrive. On that manual path, release notes are grouped by the repo's own release-please-config.json changelog sections when that file exists, so a repo declares its categories once and gets the same sectioned notes either way.

Under the hood
  • SKILL.md reads skills/commitcraft/workflows/<argument>.md and follows it; an unrecognized argument is rejected with the list of valid subcommands, never a silent fallback to commit.
  • Workflows shell out to bundled scripts via ${CLAUDE_PLUGIN_ROOT}/scripts/…, so paths resolve regardless of where the plugin cache lives.
  • commitcraft-release-detect-rp.sh — classifies release-please as FUNCTIONAL / DISABLED / ABSENT (skip flags, missing contents: write, stuck at 0.0.0); tested by bash commitcraft/tests/detect-rp.test.sh.
  • commitcraft-release-analyze.sh — semantic version analysis for the manual fallback path.
  • The PR base branch is derived via git symbolic-ref against origin/HEAD, falling back to gh repo view's default branch and then main — never hardcoded.
  • commitcraft-issues.sh — branch-based issue validation across github / linear / jira / none; the GitHub path only trusts a trailing -<num> branch suffix (e.g. fix/thing-305) as the issue number, not any number appearing elsewhere in the branch name.
  • On the manual release path, notes are grouped by the repo's own release-please-config.json changelog-sections when present, falling back to a built-in section table.
  • Conventional commits only: <type>(<scope>): <subject>, ≤50 char lowercase subject, no emoji, ≤72 char body lines — enforced by commitlint locally and in CI.
  • Files are staged individually (git add <file>), never git add -A; no attribution footers (no Co-Authored-By); --no-verify is never used — hook failures are hard stops, not bypasses.
  • The commit workflow auto-creates a feature branch when run on main.

Requirements

git, and gh (authenticated) for GitHub-issue validation, PR creation, and branch-protection provisioning — workflows degrade gracefully when gh is absent rather than blocking. The issue tracker is configurable via setup (github, linear, jira, or none); Linear/Jira read reference keys from the branch name and need no gh. Optional per-repo tooling (commitlint, gitleaks, pre-commit, release-please) is configured by the setup workflow.

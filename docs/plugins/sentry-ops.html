sentry-ops

Audit a repo's Sentry setup against the official SDK docs, work production issues through the Sentry MCP server, and add instrumentation behind a PII boundary.

Install

/plugin install sentry-ops@nautilai

What it does

One skill with four workflows over a repo's error monitoring. audit reads the repo's actual Sentry setup — SDK and version from the lockfile, Sentry.init options, integrations, sourcemap upload, release and dist wiring, sampling rates, beforeSend scrubbing, environment gating, per-runtime DSNs — then checks it against the official Sentry docs fetched at runtime rather than against a frozen checklist. triage and investigate read live issue data through the Sentry MCP server; investigate takes one issue to a root cause at file:line, navigating by tags when stack traces are minified and sourcemaps are absent. instrument adds or repairs capture sites behind a PII gate that covers both what you attach and what the SDK attaches on its own.

How it works

The audit workflow discovers before it judges. It reads the SDK and its version from the lockfile, then the Sentry.init call and everything around it — integrations, sourcemap upload, release and dist wiring, sampling rates, beforeSend scrubbing, environment gating, per-runtime DSNs. Only then does it fetch the official Sentry docs for that SDK and version through Context7 and compare. Grounding the expectations at runtime is what keeps the audit from rotting as the SDK moves.

The live half runs against the Sentry MCP server. triage lists unresolved production issues and ranks them by what is worth acting on; investigate takes one of them the rest of the way. When the stack trace is minified and no sourcemap was uploaded, it navigates by the issue's tags, read against the tagging convention the repo already uses, instead of trusting the frames.

Under the hood
  • The PII boundary covers what the SDK attaches on its own, not only what you pass it: in the browser, the page URL, referrer and headers via a default-on HTTP context integration, plus console, fetch, XHR and navigation breadcrumbs; on the server, request data including cookies, headers, query strings and bodies. Any route carrying a token or share ID in its URL becomes event data.
  • Which options gate that collection is version-dependent, so audit resolves the boundary against the docs for the SDK version the repo actually resolved rather than asserting it. Porting this section is what caught a wrong rule in the skill it came from.
  • Anything touching PII handling or sampling behavior is classified ask-user under the repo's finding-dispositions convention, never self-resolved.
  • Whether an organization slug must be passed to a Sentry MCP tool depends on whether the configured endpoint URL already bakes in the org and project — this is detected, not assumed.
  • Not every Sentry tool is directly callable; unlisted ones route through execute_sentry_tool, and search_sentry_tools resolves names and arguments.
  • Seer analysis (analyze_issue_with_seer) runs for minutes, so it is a deliberate step, not an automatic follow-up to fetching issue details.
  • Without Context7, audit falls back to WebFetch against docs.sentry.io; if that also fails it degrades to structural-only checks and states the limitation in the report rather than implying the setup was validated against docs.

Requirements

A user-configured Sentry MCP server is required for triage and investigate — they read live issue data and there is no repo-only fallback. This plugin neither bundles nor provisions it. audit and instrument read the repo only and need no MCP server. The Context7 MCP server is optional: it grounds audit in the official SDK docs, and audit degrades explicitly without it. Claude Code only — this plugin is not ported to Hermes Agent.
